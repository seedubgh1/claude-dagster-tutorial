Core Concepts

// the mental model behind Dagster

Dagster is built around Software-Defined Assets (SDAs) — you declare what data you want to produce, and Dagster handles execution order, tracking, and lineage automatically.

Primary
Asset

A persistent object in storage (table, file, model) produced by a Python function. The core unit of work.

Primary
Op

A single unit of computation. Lower-level than assets — used for tasks without a persistent output.

Execution
Job

A subset of assets or ops to execute together. Triggered manually or on a schedule.

Execution
Schedule

A cron-based trigger for a job. e.g., run every day at 9am UTC.

Config
Resource

A shared configured object — DB connections, API clients, S3 buckets — injected into assets.

Config
Sensor

A polling function that watches for external events (new files, webhooks) and triggers a job.

The asset graph

When assets depend on each other, Dagster builds a dependency graph automatically. Materializing an asset runs all upstream dependencies first.

raw_sales_csv
@asset
→
cleaned_sales
depends on raw
→
enriched_sales
joins API data
→
sales_summary
final output
Mental model: Think of assets like cells in a spreadsheet — each declares its formula (dependencies) and Dagster handles recalculation order.

Your First Asset

// the simplest possible Dagster program

A Dagster asset is a Python function decorated with @asset. Whatever it returns is stored as the asset's materialized value.

Write it manually

# my_pipeline/assets.py
from dagster import asset

@asset
def hello_world():
    """A simple greeting asset."""
    message = "Hello from Dagster!"
    print(message)
    return message

Register it

# my_pipeline/__init__.py
from dagster import Definitions, load_assets_from_modules
from . import assets

defs = Definitions(
    assets=load_assets_from_modules([assets])
)

Your First Pipeline

// chaining assets with dependencies

To create a pipeline, one asset simply receives another as a parameter. Dagster infers the dependency automatically from the argument name.

# my_pipeline/assets.py
import pandas as pd
from dagster import asset, MaterializeResult, MetadataValue

@asset
def raw_numbers():
    """Generate a list of numbers."""
    return [1, 2, 3, 4, 5, 10, 20, 30]

@asset
def filtered_numbers(raw_numbers):  # ← depends on raw_numbers
    """Keep only numbers greater than 5."""
    result = [n for n in raw_numbers if n > 5]
    return result

@asset
def number_summary(filtered_numbers):  # ← depends on filtered
    """Compute summary statistics."""
    df = pd.Series(filtered_numbers)
    return MaterializeResult(
        metadata={
            "count": MetadataValue.int(len(df)),
            "mean":  MetadataValue.float(float(df.mean())),
            "max":   MetadataValue.int(int(df.max())),
        }
    )

Ingestion

Layer 1: Data Ingestion

// reading from files and APIs with resource configuration

Claude Code prompt

"Create two Dagster assets in assets/ingestion.py. The first, raw_sales_csv, reads a CSV from a path configured via a FileSystemResource and returns a pandas DataFrame with row count as metadata. The second, raw_customers_api, fetches paginated customer data from a REST API endpoint (base URL from a resource), handles HTTP errors gracefully, and returns a DataFrame. Both use group_name='ingestion'."

What Claude should produce

# assets/ingestion.py
import pandas as pd
import requests
from dagster import asset, MetadataValue

@asset(group_name="ingestion")
def raw_sales_csv(context, file_system: FileSystemResource) -> pd.DataFrame:
    path = file_system.get_path("sales.csv")
    df = pd.read_csv(path)
    context.add_output_metadata({
        "row_count": MetadataValue.int(len(df)),
        "columns":   MetadataValue.text(str(list(df.columns))),
        "preview":   MetadataValue.md(df.head(3).to_markdown()),
    })
    return df

@asset(group_name="ingestion")
def raw_customers_api(context, api: ApiResource) -> pd.DataFrame:
    records, page = [], 1
    while True:
        resp = requests.get(
            f"{api.base_url}/customers",
            params={"page": page, "limit": 100},
            headers={"Authorization": f"Bearer {api.token}"}
        )
        resp.raise_for_status()
        data = resp.json()
        if not data["results"]: break
        records.extend(data["results"])
        page += 1
    df = pd.DataFrame(records)
    context.add_output_metadata({
        "total_customers": MetadataValue.int(len(df))
    })
    return df
Key pattern: Resources (FileSystemResource, ApiResource) are injected as parameters — Dagster handles the wiring. This makes assets easily testable: just pass mock resources in your tests.
Follow-up prompt

"Create resources.py with ConfigurableResource subclasses for FileSystemResource (with a base_dir field) and ApiResource (with base_url and token fields). Register both in Definitions inside __init__.py."

Transform

Layer 2: Transform & Enrich

// cleaning, validating, joining, and adding business logic

Claude Code prompt

"Create assets/transform.py with two assets. First: validated_sales — takes raw_sales_csv as input, drops rows with null sale_id or amount, casts amount to float, removes duplicates on sale_id, filters negative amounts, emits a data quality report as metadata showing rows dropped at each step. Second: enriched_sales — joins validated_sales with raw_customers_api on customer_id, adds a 'region' column from a mapping dict, a 'customer_segment' column (enterprise if annual_revenue > 1M else smb), and a 'quarter' column from sale_date. Use group_name='transform'."

Data quality tracking pattern

@asset(group_name="transform")
def validated_sales(context, raw_sales_csv: pd.DataFrame) -> pd.DataFrame:
    df = raw_sales_csv.copy()
    initial = len(df)

    df = df.dropna(subset=["sale_id", "amount"])
    after_nulls = len(df)

    df["amount"] = df["amount"].astype(float)
    df = df.drop_duplicates(subset=["sale_id"])
    after_dedup = len(df)

    df = df[df["amount"] > 0]
    final = len(df)

    context.add_output_metadata({
        "rows_input":         MetadataValue.int(initial),
        "dropped_nulls":      MetadataValue.int(initial - after_nulls),
        "dropped_duplicates": MetadataValue.int(after_nulls - after_dedup),
        "dropped_negative":   MetadataValue.int(after_dedup - final),
        "rows_output":        MetadataValue.int(final),
    })
    return df
Pattern
Track quality at each step

Emit metadata for rows dropped at each stage — visible in the UI's materialization history.

Pattern
Always copy inputs

Work on .copy() of upstream DataFrames — never mutate inputs in place.

Pattern
Business logic in transforms

Segmentation, region mapping, derived columns belong here — not in ingestion.

Pattern
Fail loudly

Use assert len(df) > 0 so Dagster marks the asset failed rather than silently producing empty output.

Layer 3: Aggregate & Store

// computing KPIs and writing analysis-ready output files

Claude Code prompt

"Create assets/aggregations.py with three assets in group_name='aggregations'. First: sales_by_region — takes enriched_sales, groups by region and quarter, computes total_revenue, order_count, avg_order_value, and revenue_growth_pct vs prior quarter. Second: sales_by_product — groups by product_id and quarter, same KPIs plus rank_by_revenue. Third: analysis_output — takes both aggregation assets, writes each to Parquet via FileSystemResource, returns a summary dict with file paths and row counts as metadata."

Quarter-over-quarter growth pattern

def add_qoq_growth(df, group_col, metric_col):
    df = df.sort_values([group_col, "quarter"])
    df["prev_quarter"] = df.groupby(group_col)[metric_col].shift(1)
    df["revenue_growth_pct"] = (
        (df[metric_col] - df["prev_quarter"]) /
         df["prev_quarter"] * 100
    ).round(2)
    return df.drop(columns=["prev_quarter"])

@asset(group_name="aggregations")
def sales_by_region(enriched_sales: pd.DataFrame) -> pd.DataFrame:
    agg = (enriched_sales
        .groupby(["region", "quarter"])
        .agg(
            total_revenue  = ("amount",  "sum"),
            order_count    = ("sale_id", "count"),
            avg_order_value= ("amount",  "mean"),
        )
        .reset_index()
    )
    return add_qoq_growth(agg, "region", "total_revenue")
Why Parquet? Columnar, compressed, and readable by pandas, Spark, DuckDB, Polars, and every major BI tool. Downstream analysts need zero Dagster knowledge to use these files.

Prompting Patterns

// prompts that produce reliable, idiomatic Dagster code

Pattern 1
Specify file + function name upfront

Always say "In assets/transform.py, create a function called validated_sales" — Claude generates cleaner code with less boilerplate when the destination is explicit.

Pattern 2
Name your inputs explicitly

Say "takes raw_sales_csv: pd.DataFrame as input" rather than "takes the raw data". Claude needs exact asset names to generate correct dependency wiring.

Pattern 3
Ask for metadata by default

Add "emit row counts, column names, and a 3-row preview as asset metadata" to every prompt. This makes debugging runs dramatically easier.

Pattern 4
Request group_name

Always specify group_name — it organises your asset graph in the UI so ingestion / transform / aggregation layers are visually separated.

Pattern 5
Ask for tests alongside code

Append "also generate a pytest test in tests/test_assets.py that calls this asset with a small in-memory DataFrame" to get unit tests automatically.

Pattern 6
Iterate with pasted context

Paste current code back: "Here is my current validated_sales: [code]. Now add an asset check that fails if more than 30% of rows are dropped."

Debugging Tips

// common errors and how to resolve them efficiently

Error type
Missing dependency / import error

Paste the full traceback to Claude: "I got this error materializing validated_sales: [error]. Here's the asset code: [code]. Fix it." Claude resolves missing imports and misspelled asset names reliably.

Error type
Asset not appearing in UI

Usually a missing Definitions registration. Prompt: "My new asset cleaned_data in assets/transform.py isn't in the Dagster UI. Here's my __init__.py: [code]. What's missing?"

Error type
Circular dependency

Dagster raises DagsterInvalidDefinitionError. Prompt: "I get a circular dependency error. Here are my assets with their dependencies: [list]. Suggest how to restructure them."

Error type
Slow pipeline / memory issues

Prompt: "My enriched_sales asset takes 10 minutes on 2M rows. Here's the code: [code]. Rewrite it to use pandas merge efficiently and process in chunks if needed."

The debugging loop

1
Copy the full error from Dagster UI

The run log shows the complete traceback — copy everything from the red failure line down.

2
Paste error + relevant code to Claude Code

Include the failing asset's full code, not just the error message. Context matters.

3
Apply the fix and re-materialize

Dagster auto-reloads on file save — just click Materialize again in the UI.

4
Check metadata in the UI

After a successful run, review the metadata panel to confirm data quality looks right.

Tip: Run DAGSTER_HOME=~/.dagster dagster dev to persist run history between sessions. Compare metadata from today vs yesterday to catch regressions early.
